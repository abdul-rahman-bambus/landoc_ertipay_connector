# Payment Provider: Ertipay for Odoo 18 Community

`payment_ertipay` adds an Ertipay UPI payment provider for Odoo 18 Community. It is intended for the Landoc document-services workflow in Tamil Nadu, where a customer receives an Odoo website quotation, signs/accepts it, chooses Ertipay during checkout, and completes payment through a UPI flow similar to Razorpay.

## Business purpose

The connector supports this expected flow:

1. The Odoo bot or WhatsApp bot creates/sends a standard Odoo quotation link to the customer.
2. The customer opens the quotation on the Odoo website.
3. The customer accepts/signs the quotation.
4. The customer selects **Ertipay** as the payment provider.
5. Odoo creates an Ertipay UPI payment request.
6. Ertipay returns a UPI intent link.
7. The customer pays using a UPI application.
8. Ertipay sends a secure callback to Odoo.
9. Odoo marks the payment transaction as paid, pending, or failed.
10. Existing automation, such as the n8n/WhatsApp bot, can react to the Odoo payment status update.

## Technical flow

### Authentication

Ertipay token endpoint:

```text
POST https://payin.ertipay.com/uat/token
POST https://payin.ertipay.com/prod/token
```

The module sends:

```json
{
  "email": "<configured email>",
  "apiPayinApiSecret": "<configured API secret>"
}
```

The JWT bearer token is cached on the payment provider record and refreshed before expiry to avoid the Ertipay rate limit of 5 token generations per hour.

### UPI payment creation

Ertipay UPI endpoint:

```text
POST https://payin.ertipay.com/uat/upi
POST https://payin.ertipay.com/prod/upi
```

Plain payload generated by Odoo before encryption:

```json
{
  "type": "MOB",
  "vpa": "<merchant VPA>",
  "initMode": "04",
  "txnRefId": "<Odoo transaction reference>",
  "txnAmt": "1.00",
  "txnRemarks": "<Odoo transaction reference>",
  "refUrl": "https://your-odoo-domain.com/payment/ertipay/return"
}
```

The plain payload is encrypted with AES-128-CBC and sent as:

```json
{
  "data": "<iv_hex>:<encrypted_hex>"
}
```

### Encryption

The connector implements the Ertipay-provided Node.js encryption logic in Python:

- Algorithm: `AES-128-CBC`
- Key: 32-character hexadecimal Pay-In encryption key
- IV: random 16 bytes per encryption
- Input: UTF-8 JSON string
- Output: `<iv_hex>:<encrypted_hex>`

### Callback verification

Ertipay callback route in Odoo:

```text
POST /payment/ertipay/callback
```

The controller expects:

- AES-encrypted payload in `data` or `encryptedData`
- `x-erti-signature`
- `x-erti-timestamp`

The signature is verified using HMAC SHA256:

```text
HMAC_SHA256(JSON.stringify(encryptedData) + timestamp, Pay-In Encryption Key)
```

Callbacks older than 5 minutes are rejected to reduce replay risk.

### Status mapping

| Ertipay status | Meaning | Odoo state |
| --- | --- | --- |
| `S` | Success | Done |
| `D` | Deemed Success | Pending |
| `F` | Failure | Error |

## Installation

1. Copy this repository into the Odoo addons path.
2. Ensure the Odoo server has the `openssl` command available. Most Linux servers include it by default.
3. Restart Odoo.
4. Update the app list.
5. Install **Payment Provider: Ertipay**.

## Configuration

Go to:

```text
Accounting / Invoicing → Configuration → Payment Providers → Ertipay
```

Configure:

| Field | Description |
| --- | --- |
| Merchant ID | Ertipay merchant ID, for example `ERTIxxxxxx` |
| Pay-In Email | Email/username provided by Ertipay |
| Pay-In API Secret | Token API secret provided by Ertipay |
| Pay-In Encryption Key | 32-character hexadecimal AES key provided by Ertipay |
| Merchant VPA | VPA provided during Ertipay onboarding |
| Channel Type | Usually `MOB` for UPI intent flow |
| Initiation Mode | Usually `04` for intent flow, `01` for dynamic QR if supported |

Use **Test Mode** for UAT and **Enabled** for production.

## Testing checklist

1. Configure Ertipay in test mode.
2. Use Ertipay UAT credentials and VPA.
3. Create a quotation from Odoo Sales.
4. Send the quotation link to the customer/test user.
5. Accept/sign the quotation from the website portal.
6. Choose Ertipay on the payment page.
7. Confirm that Odoo opens the Ertipay UPI intent payment screen.
8. Complete the UPI payment in UAT.
9. Verify that Ertipay calls `/payment/ertipay/callback`.
10. Confirm that the Odoo transaction becomes `done` for status `S`.
11. Confirm that the existing bot/n8n automation receives the payment status update if installed.

## Notes and limitations

- The current implementation focuses on UPI intent payments.
- Desktop users may need to continue payment from a mobile device because `upi://` links are mobile-app oriented.
- Dynamic QR support can be extended if Ertipay returns a QR URL for `initMode = 01`.
- Refund handling is not included in the initial version.
- Do not paste production credentials into source code. Store them only in the Odoo payment provider configuration.
- This module depends only on Odoo `payment`; `website_sale` is intentionally not required because quotation preview/payment links can use the standard payment flow without installing the ecommerce shop module.
